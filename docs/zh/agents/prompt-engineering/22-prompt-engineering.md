# 22 · 提示词工程 · 沟通范式之道

**学习状态**：已完成 · 2026年4月1日 · 苏格拉底式对话推导

---

## 一、核心定义
> 精心设计给AI的指令，通过明确身份、上下文、内容和输出格式，获得更准确、更符合预期结果的技巧与思维方式，就是 **Prompt Engineering（提示词工程）**。

---

## 二、从真实场景出发
### 面试网（mianshiwang）案例
AI驱动的简历分析产品，用户上传简历，AI给出分析建议。要让AI扮演好"资深HR/面试官"的角色，需要告诉它三件事：
1. 它是谁（身份）
2. 它要做什么（任务）
3. 它应该怎么输出（格式）

这三件事加在一起，就是给AI的"指令"，也就是Prompt的雏形。

### 范式的共通性
- 简历本身是固定的内容模板（基本信息、教育经历、工作经历、项目经验...）
- 面试是通向offer的固定范式
- **Prompt，同样是一种与AI沟通的范式**

---

## 三、角色区分：System vs User
```javascript
const response = await openai.chat.completions.create({
  messages: [
    { role: "system", content: "你是企业负责人，有N年经验，请从以下维度分析简历..." },
    { role: "user", content: "请分析这份简历：[简历内容]" }
  ]
})
```

| 角色 | 内容 | 特点 | 职责 |
|------|------|------|------|
| `system` | 身份设定 + 输出格式 | 固定、稳定、开发者控制 | 定义AI的行为边界和输出规范 |
| `user` | 每次上传的简历内容 | 变化、每次不同、用户输入 | 提供需要处理的实际材料 |

**单一职责原则**：就像路由进行策略区分，构造更合理的提示链，从而决定是并行化还是串行来处理信息，以便更好更快地输出内容。
> 修改分析风格 → 只改System，内容不动。

## 四、Prompt 完整结构
经过推导，一个完整的 Prompt 由四个部分构成：
```
完整 Prompt =
  Role    / 身份设定   → 决定视角和专业度
+ Context / 上下文     → 缩小范围，贴近真实场景
+ Content / 核心内容   → 要处理的实际材料
+ Format  / 输出格式   → 结构化，方便产品使用
```

### Context 的本质
同样一个问题，加了上下文，结果完全不同：
- `"我应该学AI吗？"` → AI面对全宇宙所有人，答案泛泛而谈
- `"2026年，作为前端开发的我，应该学AI吗？"` → AI面对具体的人、具体的时间点，答案精准有力

**Context 的核心作用**：把一个无限宽泛的问题，收窄到一个具体的场景里。

## 五、易经映射
$$\text{Prompt Engineering = 兑卦，沟通范式之道}$$
兑卦的核心是"丽泽兑，君子以朋友讲习"，两泽相连，互相交流，象征沟通与对话。提示词工程本质是人与AI之间的沟通范式设计，通过清晰的指令，建立人与AI之间高效的对话通道，让AI能够准确理解人的意图，输出符合预期的结果。

兑卦讲究"和悦、适中、有度"，提示词工程同样如此：既要有清晰的指令，又不能过度限制AI的创造力，在规范和自由之间找到平衡。

## 六、思考与感悟
我们既要看到提示词工程带来的便利，也要知道我们舍弃和丢弃了什么：
- ✅ 提示词工程是技巧：能让我们在与AI协作时得到更准确、高效的结果
- ⚠️ 过度依赖设限：为了追求"正确"、高效的答案，设限的同时也是在局限AI和我们自身

世界是多元的，普通视角、跨界视角同样有意义和价值。正因如此，世界才需要连接，才如此美妙。
