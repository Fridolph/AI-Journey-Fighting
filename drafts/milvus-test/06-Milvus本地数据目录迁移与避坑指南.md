# 第 06 章：Milvus 项目内数据目录与避坑指南

## 背景

当前 Milvus 的 Docker Compose 直接把数据挂载到仓库根目录下的 `./volumes`。

这种方式在 demo 阶段很方便，但也有两个需要明确认识的点：

- 仓库里的 `volumes/` 很容易在清理目录、重装环境、误删无关文件时一起被删掉
- Git 虽然会忽略这些数据目录，但“忽略提交”不等于“自动备份”
- 一旦本地目录删掉，Milvus、MinIO、etcd 相关数据通常也就一起丢了

这次就属于一个典型踩坑：容器能重新启动，但底层挂载目录已经不是之前那份数据了。

## 当前目录结构

现在 `milvus-standalone-docker-compose.yml` 使用项目内目录：

```bash
./volumes
```

实际对应挂载如下：

- `etcd` -> `./volumes/etcd`
- `minio` -> `./volumes/minio`
- `milvus standalone` -> `./volumes/milvus`

这样之后执行：

```bash
docker compose -f ./milvus-standalone-docker-compose.yml up -d
```

默认就会把数据写到当前项目目录里。

## 为什么目前仍然保留项目内目录

对于当前这个学习仓库，数据放在项目里也有实际好处：

1. 查找简单

代码、脚本、实验数据都在同一个目录层级里，学习阶段更容易理解和排查。

2. 迁移方便

整个项目打包、移动到另一台机器时，不需要额外找数据目录。

3. 结构直观

看到 `volumes/` 就知道这就是本地 Milvus 数据，不容易搞不清路径。

## 当前管理建议

建议把思路定成下面这样：

- `volumes/` 放项目内，便于学习和日常调试
- `.gitignore` 明确忽略 `volumes/`，避免误提交大文件和中间件数据
- `backups/` 也一并忽略，用脚本做阶段性备份
- 原始数据源、建库脚本、写入脚本必须保留，确保丢库后可重建

## 备份建议

对学习项目来说，不一定需要复杂备份系统，但至少建议做这两层：

1. 保留原始数据源

例如电子书、分块脚本、embedding 写入脚本、collection schema，确保库丢了以后还能重建。

2. 定期备份项目内目录

最简单的方法就是定期备份 `./volumes`。

```bash
./scripts/milvus-backup.sh
```

如果希望备份更稳一点，可以在停掉容器后执行：

```bash
./scripts/milvus-backup.sh --safe
```

这样会先停容器、打包 `volumes/`、再自动拉起服务。

## 我自己的结论

对于当前这个学习仓库，更合适的策略是：

- 代码、文档、Milvus 数据先放在同一个项目里，降低理解成本
- `volumes/` 和 `backups/` 一律忽略提交
- 原始数据源和重建脚本必须保留
- 每完成一个重要实验阶段就手动备份一次

这样更符合“可复现 + 可理解 + 可迭代”的学习目标，也更不容易因为一次误删把实验环境整个打回起点。
