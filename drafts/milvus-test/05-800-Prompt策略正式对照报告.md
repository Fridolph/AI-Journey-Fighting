# 第 05 章：`c800` Prompt 策略正式对照报告

## 一、实验目标

这轮实验的目标不是比较分块参数，而是：

> 在 **同一组召回结果** 下，只调整回答阶段的 prompt 策略，观察回答质量会如何变化。

也就是说，这一轮只测：

- `strict`：严格贴证据
- `balanced`：证据优先 + 有限推断
- `inferential`：较积极推断

不改：

- chunkSize
- chunkOverlap
- topK
- 问题集

## 五、核心发现

## 1）`balanced` 是当前最适合的默认策略

从 10 题整体看，`balanced` 的优势最稳定，尤其在下面两类题里最明显：

### A. 证据已经够，但需要“组织能力”的题

例如：

- Q4 化功大法特点
- Q6 六脉神剑时灵时不灵
- Q7 鸠摩智为什么一定要得到六脉神剑

这类题不是召回问题，而是回答层要不要把多条证据串成一个更完整的解释。

`strict` 的问题是：

- 往往只敢逐条复述
- 不太愿意把分散证据整合成“原因链”

`inferential` 的问题是：

- 虽然信息量更大
- 但有时会把“合理猜测”写得过满

`balanced` 刚好处于中间：

- 不乱编
- 也不只是机械复述
- 会把“直接证据”和“有限推断”分开写

这很适合真实产品中的默认回答风格。

---

## 2）对“是否有明确结论”的题，`strict` 反而更强

最典型的是：

- Q2：小说里有没有明确封谁是“天下第一”？

这个问题本质上不是“猜谁最强”，而是：

> 原文有没有明确写死？

所以这题里：

- `strict` 直接答“没有明确封谁是天下第一”，最清楚
- `balanced` 虽然也没错，但开始多讲“段誉更可能”
- `inferential` 更容易把题目从“有没有明确封”带偏成“谁更像”

这说明一个很重要的产品经验：

> **不是所有题都适合鼓励推断。**

如果用户的问题本身是在问“原文是否明确写了”，那就应该优先给明确定性回答，而不是抢着做候选排序。

---

## 3）召回不对时，prompt 再好也救不了

最明显的是：

- Q1：谁的武功最强
- Q3：扫地僧在少室山一战中做了什么
- Q8：阿朱是怎么死的

这些题的共同点是：

- 当前 `c800` 召回片段并没有把真正关键证据召回来
- 或者召回只召到外围议论，而没召到核心情节

这时三种 prompt 的区别就变成：

- `strict`：老老实实说“证据不足”
- `balanced`：能多解释一句“缺的是哪一类证据”
- `inferential`：最容易开始在错误候选集合里做扩展

这再次证明：

> **回答层优化不能替代召回层优化。**

尤其像“谁最强”“怎么死的”“做了什么”这种强事实题，如果关键片段没进来，回答层越主动，越可能把用户带偏。

---

## 4）`inferential` 不是没用，但不适合做默认

`inferential` 在本轮实验里不是完全失败。

它的价值主要在：

- 能给出更完整的候选解释
- 在探索式问答里信息量更大
- 对用户“只想先听一个大概判断”的场景更友好

但问题也很明显：

- Q1 容易把“当前片段下的候选”写成“仿佛全书定论”
- Q2 会从“有没有明确封”滑向“谁更像”
- Q8 这类召回缺失题里，越积极越容易说多

所以它更适合：

- 内部实验模式
- 原型验证阶段
- 或者显式告诉用户“以下是基于当前证据的推测”

而不适合直接作为默认答案模式上线。

## 六、按题型看，三种策略各自适合什么

### 1）事实核验题

例如：

- 有没有明确封谁是天下第一
- 阿朱是怎么死的
- 扫地僧做了什么

推荐：

- 首选 `strict`
- 次选 `balanced`
- 不建议默认 `inferential`

原因：

- 这类题最怕“明明没证据，还往前猜”

### 2）原因归纳题

例如：

- 六脉神剑为什么时灵时不灵
- 鸠摩智为什么要得到六脉神剑
- 化功大法有什么特点

推荐：

- 首选 `balanced`

原因：

- 这类题往往不是片段里一句话写死
- 需要把多个证据点组织成一条解释链

### 3）人物概括 / 列举题

例如：

- 三兄弟各自最典型的武功是什么
- 哪些人物轻功特别突出

推荐：

- 首选 `balanced`

原因：

- 用户更在意“给我一个可用答案”
- `strict` 容易因为一个人物缺明文就整体变得保守
- `balanced` 更适合做“片段内候选归纳”

## 七、这轮实验最终结论

### 正式结论

在 `c800 / topK=5 / 同一组召回结果` 的条件下：

> **`balanced` 是当前这个《天龙八部》RAG 项目里最适合作为默认回答策略的 prompt 模式。**

原因有三点：

1. **整体平均分最高**
2. **在“有限推断题”上明显优于 strict**
3. **又没有 inferential 那么容易越界**

### 三种策略的建议定位

- `strict`
  - 适合作为高保守模式
  - 适合“是否有明确证据”类问题
  - 适合作为 fallback / 安全兜底

- `balanced`
  - 适合作为默认产品模式
  - 适合大多数知识问答、RAG 问答、简历问答
  - 能兼顾正确性、可用性、解释性

- `inferential`
  - 适合作为实验模式
  - 适合探索型问答
  - 不建议默认开启

## 八、对后续产品设计的启发

这轮实验说明，后续在 `my-resume` 或其他项目里，回答层最好拆成三层心智：

### 1）直接结论层

先说：

- 能不能回答
- 当前最直接的结论是什么

### 2）证据层

再说：

- 这个结论来自哪些片段

### 3）有限推断层

最后再说：

- 当前片段下，更可能是谁 / 为什么
- 这部分是推断，不是原文明写

如果做成产品交互，我建议把这一层显式标出来，比如：

- `直接证据`
- `有限推断`
- `证据不足的部分`

这样用户对回答的信任感会更强，也更容易接受“为什么这次只能回答到这里”。

## 九、下一步建议

下一步我建议继续做两件事：

### 1）把 `balanced` 固化成默认 prompt 模板

并在回答结构里强制输出：

- 结论
- 直接证据
- 有限推断
- 不足之处

### 2）专门补一轮“召回失败题”的优化实验

重点挑这几题：

- Q1 谁的武功最强
- Q3 扫地僧在少室山一战中做了什么
- Q8 阿朱是怎么死的

因为这几题目前主要不是 prompt 问题，而是召回没把核心片段打进来。

也就是说，后面真正该优化的是：

- query 改写
- topK
- rerank
- 多轮检索
- 关键词 + 向量混合检索

---

## 十、一句话总结

这轮 `c800` prompt 对照实验最重要的结论不是“模型会不会推理”，而是：

> **当召回固定时，回答层 prompt 足以显著改变“有用性”；但只有 `balanced` 能在“可用”与“不乱猜”之间取得最好的产品平衡。**
