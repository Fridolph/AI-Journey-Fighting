# my-resume：简历 RAG 落地开发建议

## 背景

这一份建议不是从空想出发，而是基于两部分已经完成的学习与验证：

1. `examples/resume-memory-rag-qa`
   - 已经做了从简历解析、Milvus 入库、问答检索，到多轮版本演进的一整套实验
2. `drafts/resume-memory-rag-qa`
   - 已经沉淀了字段设计、检索优化、管线拆分、主证据优先、跨区补证等阶段总结

当前目标不是把 demo 原样搬进 `my-resume`，而是：

- 提炼出真正稳定、可迁移的能力
- 先解决正式项目里最关键的结构问题
- 再逐步引入更复杂的问答与策略优化

---

## 当前判断

`my-resume` 目前已经具备：

- 基础 RAG 模块
- Milvus 接入能力
- 检索路由、重排、上下文构建的基础设施
- 简历同步到检索索引的入口

但当前简历同步仍然偏粗粒度：

- 简历导出为 markdown
- 直接以整份内容作为单个 chunk 入索引

这意味着：

- 可以检索
- 但检索粒度较粗
- 很容易出现“能答，但不够准”“证据不聚焦”的问题

所以，`my-resume` 当前最优先该升级的，不是 Prompt 花样，而是：

> **简历知识如何被组织进向量库。**

---

## 从 demo 中已经验证过的关键结论

### 1. 简历不适合整篇粗切入库

简历天然是强结构化内容：

- 基本信息
- 核心竞争力
- 专业技能
- 工作经历
- 项目经历

如果整份 markdown 直接切块：

- 问题一来，召回结果容易混杂多个主题
- 很容易命中“沾边但不是最关键”的内容
- 最终回答会显得泛化、啰嗦、不聚焦

更合理的做法是：

- 先按业务语义拆成记录
- 再只对过长记录做二次 chunk

### 2. 经验类问题不能让 `skills` 抢主位

纯向量召回会天然偏爱：

- 标题高度相似
- 关键词密集
- 表达很“标准答案”的 `skills` 片段

但在简历问答里，用户很多时候真正想问的是：

- 做过什么
- 在哪做过
- 哪个项目最能代表你

这类问题更应该优先依赖：

- `projects`
- `work_experience`
- 某些 `core_strengths`

所以正式项目里一定要引入“问题类型意识”和“证据优先级”。

### 3. `topK` 调大只能补召回，不能单独解决质量问题

调大 `topK` 的好处是：

- 本来没进候选池的重要项目，有机会被捞上来

但它不能自动解决：

- 哪些该排前面
- 哪些是噪音
- 哪些该真正进入 Prompt

所以正式项目里应该坚持这条顺序：

1. 先让候选别漏
2. 再做业务重排
3. 最后做裁剪和上下文组装

### 4. 会话历史和简历知识不能混存

简历块是：

- 候选人的原始事实知识

会话历史是：

- 某个用户在某个 session 下的上下文状态

两者职责完全不同，不能混进同一个 vector collection。

如果混存，会出现：

- 语义污染
- 调试困难
- AI 容易引用自己之前说过的话，而不是原始事实

### 5. 阈值不是通用真理

在 demo 中调出来的：

- `skills` 降权幅度
- `noise threshold`
- `cross_support` 门槛
- `reserve` 最低分

这些都不是正式项目可以无脑照搬的常数。

它们只在以下前提下成立：

- 当前简历数据
- 当前 embedding 模型
- 当前向量分布
- 当前问题类型

所以迁移时应该：

- 迁思路
- 不迁死数值

---

## 对 my-resume 的总体建议

### 总原则

先升级“知识组织方式”，再升级“回答策略”。

换句话说：

- 第一阶段解决“简历怎么入库更合理”
- 第二阶段解决“怎么检索得更准”
- 第三阶段再解决“怎么回答得更像简历顾问”

不要一开始就把 demo 里的所有高级策略一次性搬进正式项目。

---

## 建议的目标形态

### 方案概览

建议把 `my-resume` 的简历 RAG 能力拆成两条并存的索引思路：

1. `resume_core_document`
   - 保留全文级简历文档
   - 用作兜底召回

2. `resume_semantic_chunks`
   - 新增语义字段块索引
   - 用作主检索来源

这样做的好处：

- 不会丢掉“整份全文视角”
- 也能获得细粒度证据召回能力

### 推荐字段

建议在语义块中优先保留这些字段：

- `source_id`
- `source_scope`
- `source_version`
- `locale`
- `section`
- `subsection_key`
- `subsection_title`
- `entity_type`
- `title`
- `content`
- `tags`
- `chunk_index`
- `chunk_count`
- `content_hash`
- `embedding`

### 字段职责建议

- `section`
  - 一级业务分区，如 `skills`、`projects`

- `subsection_key`
  - 程序友好的稳定标识，如项目名 slug、公司名 slug

- `subsection_title`
  - 给人读的标题，如 `my-resume`、`GreenSketch`

- `entity_type`
  - 内容形态，如 `skill_item`、`project_summary`、`project_detail`

- `title`
  - 当前记录的展示标题

- `content`
  - 真正参与 embedding 的正文

- `tags`
  - 多维辅助标签，用于检索补充、过滤、日志解释

## 正式项目建议分阶段落地

### Phase 1：先升级简历同步链路

### 目标

把当前“整份 markdown 一块入索引”，升级成“语义字段块入索引”。

### 建议先改什么

- `resume-rag-sync.service.ts`
  - 从当前的单 chunk 写法，改为“多 record / 多 chunk 写入”

- 新增 resume 专用 parsing / chunking 模块
  - 不建议直接复用 `user_docs` 固定窗口切块逻辑
  - 需要一套简历专用的语义拆分逻辑

### 为什么先做这一步

因为正式项目最先要保证的是：

> **知识进入向量库时，就是适合被检索的形态。**

如果这一层还是粗的，后面再复杂的 Prompt 和重排也救不回来。

### 这一阶段先不要做的事

- 不要先上复杂多轮
- 不要先上 cross-section support
- 不要先引入大量模板
- 不要先调很多实验阈值

---

### Phase 2：引入简历场景的检索与重排

### 目标

让经验类问题不再被 `skills` 抢主位。

### 建议做法

先只区分少量问题类型：

- `experience`
- `skill`
- `profile`

对应优先 section：

- `experience`
  - `projects`
  - `work_experience`
  - `core_strengths`

- `skill`
  - `skills`
  - `core_strengths`

- `profile`
  - `profile`
  - `core_strengths`

### 这一阶段的重点

不是做复杂智能分类，而是先建立：

- “不同问题，不同 section 优先级” 这层基础意识

---

### Phase 3：升级上下文组装与回答策略

### 目标

让系统从“能答”升级到“答得更像简历顾问”。

### 建议做法

先引入简单版本的证据分层：

- `primary evidence`
- `support evidence`

经验类问题：

- `primary`
  - 优先选 `projects / work_experience`

- `support`
  - 允许保留高相关的 `skills / core_strengths`

### Prompt 建议

第一阶段只保留两套模板即可：

- `resume_general_qa`
- `resume_experience_qa`

Prompt 要求建议固定三条：

1. 只基于给定证据回答
2. 证据不足时明确说明
3. 先讲主证据，再补辅助信息

---

### Phase 4：产品化能力再逐步补上

### 可以后续再加的内容

- session store
- 流式输出
- 检索过程日志
- 答案质量评估
- 后台可视化调试面板
- 独立 history retrieval

这些都重要，但不应该抢在前面做。

---

## 不建议直接照搬的内容

以下内容不建议直接从 demo 原样复制到 `my-resume`：

### 1. v6 / v7 的全部阈值

原因：

- 它们是实验参数
- 不是正式项目真理

### 2. CLI 风格的参数和输出组织

原因：

- demo 为了实验方便
- 正式项目要服务 API / 管理后台 / 用户交互

### 3. 会话历史写入向量库

原因：

- 当前阶段没有必要
- 会混淆知识库与记忆库职责

### 4. 过度抽象的多版本并存结构

原因：

- demo 有多版文件，是为了学习演进
- 正式项目里应该直接提炼成一套当前最稳的结构

---

## 需要直接吸收的工程经验

以下几条建议，应该直接进入 `my-resume` 的开发规范：

### 1. Embedding 配置和 Chat 配置分离

必须明确区分：

- `LLM model`
- `Embedding model`
- `baseURL`

切换供应商时，三者都要重新核对。

### 2. 向量维度不能手写假定值

应该：

- 先实际请求 embedding
- 读取真实返回维度
- 再创建或校验 collection schema

### 3. 不要只看 `insert_cnt`

写入后至少检查：

- `status`
- `reason`
- `flush`
- `row_count`

### 4. 主键不能只靠标题

应该保证：

- 稳定
- 唯一
- 最好可追溯

推荐：

- 结构前缀 + 内容 hash

### 5. 中文字段长度按字节风险预留

尤其：

- `tags`
- `title`
- `subsection`

不要只按 JS `length` 想当然处理。

### 6. 知识库与会话状态严格分离

这是后续所有多轮与记忆能力的前提。

---

## 建议的实施优先级

如果只给 `my-resume` 选一个最先开始的改造点，我建议是：

### 第一优先级

> 把 `resume-rag-sync` 从“整份简历单块入库”升级成“语义字段块入库”。

### 第二优先级

> 在现有 RAG 检索链路中加入简历场景的 section 优先级与轻量重排。

### 第三优先级

> 增加 `primary / support` 证据组织和两套基础 Prompt 模板。

### 第四优先级

> 再考虑 session、流式输出、历史语义检索等扩展能力。

---

## 我的最终建议

对 `my-resume` 来说，当前最值得迁移的，不是某个 ask 脚本的最终形态，而是这套方法论：

1. 先把简历拆成更适合检索的知识形态
2. 再按问题类型组织检索优先级
3. 再把证据分层后喂给模型
4. 最后才是体验层和多轮层优化

一句话总结：

> **先把“知识怎么进库”做对，再把“答案怎么说得更好”做好。**
