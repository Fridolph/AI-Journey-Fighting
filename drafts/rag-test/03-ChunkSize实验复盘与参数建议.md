# 第 03 章：Chunk Size 实验复盘与参数建议

## 实验目标

验证不同切块参数对 RAG 检索质量与回答风格的影响，重点观察：

- 回答是否更贴近原文证据
- 是否出现“过度发挥”
- 不同参数下的检索片段差异

## 实验样本

- `examples/rag-test/tests/rag-test-base.md`
- `examples/rag-test/tests/rag-test-small.md`
- `examples/rag-test/tests/rag-test-large.md`

统一问题：

> 父亲的去世对作者的人生态度产生了怎样的根本性逆转？

## 参数对照

### 1) Base

- `chunkSize: 500`
- `chunkOverlap: 50`
- 分块数：4

### 2) Small

- `chunkSize: 150`
- `chunkOverlap: 30`
- 分块数：16
- 额外配置：`batchSize: 10`（embedding 服务单批上限）

### 3) Large

- `chunkSize: 1000`
- `chunkOverlap: 80`
- 分块数：更少（大块语义更完整）
- 额外配置：`batchSize: 10`

## 结果观察

## 1) 三组都能回答核心问题

说明在当前语料规模较小（单篇文章）且问题较宽泛的情况下，三组参数都能召回关键证据，不会出现明显“答非所问”。

## 2) Small 的“保守回答”是本次关键收获

`small` 版本里出现了类似：

> 原文未明确使用“根本性逆转”这一强表述……

这说明模型在更细粒度证据下更容易“贴证据说话”，对问题里的强表述会更谨慎，减少过度推断。

## 3) Large 更完整，但有时更“顺滑叙事”

`large` 往往能拿到更长上下文，回答更完整、连贯；但也更容易把相邻语义一并吸收，出现“合理但不够克制”的扩展描述。

## 4) Base 是可用折中

在当前样本下，`base` 在“上下文完整度”和“检索粒度”之间比较均衡，属于开箱即用的安全参数。

## 为什么会这样（原理解释）

- `chunk` 越小：检索粒度越细，证据定位更精准，但上下文碎片化风险更高
- `chunk` 越大：上下文更完整，但噪声也更容易一起进 prompt
- `overlap`：用于缓解句子被切断导致的语义丢失

所以参数本质是在平衡：

> 证据精确度 vs 语义完整度

## 本次实验结论

1. `chunk` 调参不是“越小越好”或“越大越好”，而是场景平衡。
2. 当前样本下三组总体差异不大，但 `small` 的“证据克制”很有价值。
3. RAG 质量评估不应只看“回答流畅度”，更要看“是否忠于检索证据”。
4. 服务商有 embedding 批量限制时，需要显式设置 `batchSize`（本次为 10）。

## 实战参数建议（当前项目阶段）

- 首选起步参数：`chunkSize=400~600`，`chunkOverlap=40~80`
- 若答案经常“泛化过度”：适度减小 `chunkSize`
- 若答案经常“语义断裂”：适度增大 `chunkSize` 或 `overlap`
- 大文档/多分块时，记得限制 embedding `batchSize`

## 给后续博客可直接引用的一句话

> 这次实验最大的收获不是“哪组参数绝对更好”，而是确认了：RAG 的优化目标应优先追求“基于证据的克制回答”，而不是“看起来更会说”。
