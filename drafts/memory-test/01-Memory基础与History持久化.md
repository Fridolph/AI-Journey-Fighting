# 第 01 章：Memory 基础与 History 持久化

## 本章目标

- 理解为什么大模型本身是“无状态”的
- 理解 messages / history 为什么就是最基础的 memory
- 跑通短时记忆（内存）与长时记忆（文件持久化）
- 建立一个简单但重要的认识：Memory 不是“魔法记忆”，本质是你把什么上下文重新喂给模型

## 对应源码

- `examples/memory-test/src/history-test.mjs`
- `examples/memory-test/src/history-test2.mjs`
- `examples/memory-test/src/history-test3.mjs`

## 这章在讲什么

大模型本身是无状态的。  
所谓“它记得上一次说了什么”，本质不是模型真的记住了，而是你把之前的消息历史再次传给了它。

也就是说：

- 模型本身不记得
- 你的程序替它“保留上下文”
- 然后在下一轮调用时再一起发出去

这就是最基础的 Memory。

## 先建立一个核心心智模型

之前我们在 Agent/tool 的循环里一直在做这件事：

1. `SystemMessage` 说明角色
2. `HumanMessage` 记录用户问题
3. `AIMessage` 记录模型回答
4. `ToolMessage` 记录工具结果
5. 然后把这些一起继续传给下一轮模型

所以 Memory 从来不是某个神秘能力，而是：

> **你如何保存和管理消息历史。**

## Memory 的两个维度

### 1) 存储层（history 存哪）

- 内存
- 文件
- Redis
- 数据库

### 2) 逻辑层（上下文怎么管）

- 截断
- 总结
- 检索

这章前半段主要在讲“存储层”。

## 关键代码理解

## 1) `history-test.mjs`：短时记忆（InMemory）

核心对象：

- `InMemoryChatMessageHistory`

流程很简单：

1. 创建 `history`
2. `addMessage` 加入用户消息
3. `getMessages()` 取出全部历史
4. 拼上 `SystemMessage`
5. 调 `model.invoke(messages)`
6. 再把模型返回写回 history

第二轮时，模型之所以能接着聊，是因为上一轮消息又被一起发过去了。

所以这里的“记忆”其实就是：

> **进程内保存的 messages。**

## 2) `history-test2.mjs`：长时记忆（文件持久化）

核心对象：

- `FileSystemChatMessageHistory`

这里和内存版的区别不在于调用模型逻辑，而在于：

- 消息不再只存在运行时内存
- 而是写入 `chat_history.json`

这意味着：

- 脚本结束后消息不会丢
- 下次运行还能继续恢复对话

这就是长时记忆（LTM）的雏形。

## 3) `history-test3.mjs`：从文件恢复历史继续对话

这里最重要的动作是：

- 重新创建同一个 `sessionId`
- 再调用 `getMessages()`

于是之前两轮消息就被恢复出来，然后继续第三轮提问：

- 前两轮：红烧肉怎么做 / 好吃吗
- 第三轮：需要哪些食材

模型能接上，是因为第三轮调用时，前两轮记录已经重新装回 prompt 里了。

## 你现在可以这样理解短时记忆 / 长时记忆

- **短时记忆**：保存在进程内存里，进程结束就没了
- **长时记忆**：保存在文件/数据库里，可跨会话恢复

像 Cursor 这类工具，本质上也是在做“历史持久化 + 恢复”。

## 本章真正重要的结论

1. 模型不是天然有记忆，而是程序在管理消息历史。
2. Memory 的本质是“历史消息管理”，不是“模型突然变聪明了”。
3. `history` 解决的是“消息存哪”，还没解决“上下文太长怎么办”。
4. 后半章讲的截断、总结、检索，解决的是“上下文怎么压缩和取舍”。

## 一句话总结

> 这章前半部分要记住的只有一句话：**Memory = 可被再次喂给模型的上下文历史。**
