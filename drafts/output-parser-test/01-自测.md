更准确一点可以说：
- 如果拿不到稳定的 tool chunk 结构，就退成普通流式文本/日志展示，不强求实时 JSON 视图

---

**这轮你最该记住的一句话**

不是“有结构就用 parser”，而是：

```md
固定 JSON 结果，先想 withStructuredOutput
要实时过程感，先想 stream
要实时看 tool 参数，先想 JsonOutputToolsParser
XML/YAML 这类非 JSON，才想专用 Output Parser
```

**你现在的主要短板**
不是不会选，而是第 1 题里还会把：
- “结构化输出”
- “专用格式输出（XML）”

这两件事混在一起。

以后先问自己一句：

`业务是不是明确要求 XML / YAML？`

- 如果不是，那大概率先别想专用 parser
- 如果只是固定字段给前端消费，先想 `withStructuredOutput`

**Output Parser / Tool / Structured Output 决策表**

| 场景 | 首选方案 | 为什么 | 常见降级 |
| --- | --- | --- | --- |
| 固定 JSON 字段，非流式 | `withStructuredOutput` | 最省心，结构化结果更稳，前端也最好消费 | `jsonMode` -> `StructuredOutputParser` |
| 需要边生成边展示内容 | `Output Parser + stream` | 真正有流式体验，不用等完整结构校验后才返回 | 直接纯文本 stream |
| 实时展示 tool 调用参数 | `Tool Calls 原始流 + JsonOutputToolsParser` | 能在参数未完整时尽量拼出 JSON 视图 | 普通日志流 / 文本流展示 |
| 输出 XML / YAML | `专用 Output Parser` | 这类不是 JSON，不能套普通结构化 JSON 思路 | prompt 约束 + 手动 parse |
| 第三方兼容 OpenAI，但结构化协议不稳定 | `withStructuredOutput` 先试 | 如果支持，开发成本最低 | `jsonMode` -> `Output Parser` |
| 图表、评分卡、仪表盘一次性渲染 | `withStructuredOutput` | 结构完整性比过程感更重要 | loading + 最终一次性展示 |
| 追问思路、推理过程、边思考边显示 | `Output Parser + stream` | 用户更在意“过程可见” | 纯文本流式 |

---

**一条总原则**

先分清你要的是哪一种：

1. `最终结构化结果`
2. `过程可视化`
3. `tool 参数流`
4. `非 JSON 格式`

只要这一步分清，选型基本不会乱。

---

**快速判断口诀**

```md
普通 JSON，先想 withStructuredOutput
真要流式，先想 Output Parser + stream
tool 参数流，先想 JsonOutputToolsParser
不是 JSON，而是 XML/YAML，就想专用 Output Parser
兼容性报错，先查 provider / model 能力，再想降级
```

**常见误区**

**误区 1**
“只要有结构化需求，就用 parser”

不对。
现在普通 JSON 结构化，通常先想 `withStructuredOutput`。

**误区 2**
“StructuredOutputParser 比 JsonOutputParser 高级，所以更通用”

不对。
它只是更适合“你要明确描述 JSON 字段结构”的场景，不等于它适合 XML/YAML。

**误区 3**
“用了 withStructuredOutput，再配合 stream 就有很好的流式体验”

不一定。
很多时候它会等结构完整后才给你结果，不是真正的“边生成边展示”。

**误区 4**
“报 response_format unavailable，多半是 prompt 写错了”

不对。
第一反应应该是：
`模型 / 供应商是否支持这套结构化协议`

如果你能稳定说出下面这几句，就算过关了：

1. `withStructuredOutput` 适合普通非流式 JSON 结构化输出
2. 真流式展示时，`Output Parser + stream` 往往更合适
3. tool 参数分块流要看 `JsonOutputToolsParser`
4. XML / YAML 这种非 JSON 格式要走专用 parser
5. 结构化报错先查模型兼容性，不要先怪业务代码
