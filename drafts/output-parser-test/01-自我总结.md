## 二次复盘后的纠偏

这一轮跑代码 + 小考后，最重要的不是又记住了几个 API 名字，而是把几个容易混淆的边界慢慢分开了。

### 1. JsonOutputParser 和 StructuredOutputParser 都不是“模型原生能力”

我一开始有个误区，会把 `StructuredOutputParser` 理解成“大模型本身更强的结构化能力”。

现在修正为：

- `JsonOutputParser`
- `StructuredOutputParser`

这两个都属于 **LangChain 的解析层封装**。

它们的本质思路还是：

1. 在 prompt 里告诉模型按什么格式输出
2. 再由 parser 去解析返回结果

也就是说，它们更接近“提示词约束 + 解析”这条路线，而不是模型原生协议保证。

### 2. JsonOutputParser 和 StructuredOutputParser 的差异

它们都能处理 JSON，但定位不完全一样。

#### JsonOutputParser

更像是：

- 我已经要求模型返回 JSON
- 你帮我把返回内容解析出来
- 顺手处理一些常见脏格式，比如 markdown code block

适合理解成：

> “帮我把模型输出尽量抠成合法 JSON”

#### StructuredOutputParser

比 `JsonOutputParser` 多了一层：

- 字段有哪些
- 每个字段的含义是什么
- 字段类型是什么
- 复杂对象的结构长什么样

所以它更适合理解成：

> “我不只是要 JSON，我还要你按这个 schema 返回”

### 3. 为什么普通结构化 JSON 现在更优先考虑 withStructuredOutput

这不是因为它“官方封装所以更高级”，而是因为它更偏向：

- 优先利用模型原生支持的结构化能力
- 能少写 prompt 就少写 prompt
- 能少手工 parse 就少手工 parse

所以在普通非流式场景下，如果只是想拿到稳定的结构化 JSON，优先级通常是：

1. `withStructuredOutput`
2. 不兼容时再考虑 `jsonMode`
3. 再退到 `StructuredOutputParser` / `JsonOutputParser`

### 4. 真正流式场景为什么不优先 withStructuredOutput

这也是这章最容易想错的一点。

`withStructuredOutput` 的强项是：

- 结果稳定
- 结构合法

但它的弱项是：

- 往往要等整段结构拼好、校验通过后，才给出最终结果

所以它虽然“结构化很稳”，但不一定“交互很流畅”。

如果用户特别在意：

- AI 正在思考
- 内容边生成边可见

那更自然的路线通常是：

- `Output Parser + stream`

### 5. tool 参数流和普通文本流不是一回事

这是这轮小考里我比较容易混掉的点。

如果只是：

- 边生成分析文字
- 边把内容展示给前端

那更偏：

- `Output Parser + stream`

但如果是：

- 模型正在调用 tool
- 参数是分块回来的
- 希望参数还没完整时，也尽量给前端看到当前 JSON 视图

那更偏：

- `Tool Calls 原始流 + JsonOutputToolsParser`

所以不能把：

- 普通文本流
- tool 参数流

看成一回事。

### 6. XML / YAML 这类不是 JSON 的格式，要想到专用 Output Parser

我一开始会把“有结构化要求”直接和 `StructuredOutputParser` 绑定。

现在修正为：

- `StructuredOutputParser` 更偏 JSON schema 风格
- XML / YAML 这类非 JSON 格式，要优先想到专用 Output Parser

也就是：

- `XMLOutputParser`
- 其他对应 parser

### 7. 结构化报错时，先查模型 / 供应商能力，不要先怀疑业务逻辑

这次实际踩坑最有价值的一点就是：

当看到这类错误：

```txt
This response_format type is unavailable now
```

第一反应不应该是：

- prompt 写错了
- schema 写错了
- 参数是不是少了

而应该先想：

- 当前 provider 支不支持这套协议？
- 当前模型支不支持 `json_schema`？
- 支不支持 tool calling？

也就是先判断：

> 这是“能力兼容性问题”，还是“代码使用问题”

## 1. `JsonOutputParser` 是干什么的？

你答：

这个方向对，但再精确一点：

> `JsonOutputParser` 不是“把任意自然语言自动变成 JSON”，而是用来**解析模型返回的 JSON 风格内容**，并处理一些常见脏格式，比如 markdown code block 包裹。

## 2. `StructuredOutputParser` 比它多了什么？

这里你现在的理解还混了一点，我帮你纠正一下。

你说：

### `StructuredOutputParser`
- 你不只是说“返回 JSON”

  - 字段类型是什么
  - 甚至嵌套结构长什么样
- 更像“按这个 schema 给我返回”

所以它比 `JsonOutputParser` 多的是：

- **更强的结构描述能力**
- **更明确的格式约束**

你看到“结果好像差不多”，是因为这几个 demo 本身都比较简单。  
但一旦结构复杂起来，比如嵌套对象、数组对象，`StructuredOutputParser` 的价值就明显了。
