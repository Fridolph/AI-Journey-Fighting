更准确地说，不是“因为进程是异步”，而是：

也就是说，这里更偏数据库写入可见性问题，不只是 JS 异步。

**如果整理成更标准的答案，可以这样说**

> `insert_cnt = 0` 不能直接等价于“代码没执行成功”，因为它只说明最终成功写入的数据条数是 0，不代表前面的逻辑没跑。  
> 在真实情况里，脚本可能已经完成了连接 Milvus、创建 collection、解析数据、生成向量、发起 insert 请求等步骤，只是在最终入库校验时被数据库拒绝了。  
> 所以遇到这种问题时，我现在会这样排查：
> 1. 先看日志，确认流程已经执行到哪一步  
> 2. 检查 insert 返回结果里的 `status` 和 `reason`  
> 3. 查看是不是 schema 和数据不匹配，比如向量维度不一致、字段长度超限、主键冲突、字段格式不合法  
> 4. 执行 `flush` 后再查 `getCollectionStatistics`，确认真实 `row_count`，不要只相信 `insert_cnt`  
>  
> 这次真实踩坑里，后面就发现并不是“脚本没跑”，而是某条 `tags` 内容超过了 schema 的长度限制，导致整批插入失败。
