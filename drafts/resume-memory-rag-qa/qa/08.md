## 8. 为什么版本演进里要把 Retriever / PromptBuilder / RAGPipeline 拆开？如果不拆，后面会卡在哪里？

我的理解是，单一职责，拆分后逻辑会更加清晰，也便于后续维护：

Retriever 取回，该函数 通过函数参数形式 传递了很多值，中间过程进行了封装 只返回要召回的条目；

PromptBuilder 也是函数封装，通过策略模式 取我们要选用的prompt，这样业务逻辑不会和流程逻辑搅在一起，便于解耦和维护；该函数返回的是 有 systemMessage humanMessage 组成的数组，符合通用的传参形式；

RAGPipeline 相对上面两个又复杂了一层，同函数名 把 rag 的关键步骤 进行了拆解 ~ 具体代码逻辑记不清了， 大概就是chunk split 如何做 去噪、重排等一系列逻辑都在其中；

**需要纠偏的一点**

这里有一点容易混：

  这些**在线问答步骤**串起来

而不是再去负责最前面的简历切块。

**这题真正的核心，是“不拆会卡在哪里”**
这个你答了一点，但还可以更具体。

如果不拆，后面会卡在这几个地方：

### 1. 改一个策略，要改整条主流程
比如你只想：
- 调一下 topK
- 改一下 section 权重
- 换一个 prompt 模板

结果却要进一个超长 `main()` 里改很多地方，很容易越改越乱。

### 2. 新场景会不断复制脚本
比如后面想支持：
- 简历问答
- 岗位匹配
- 技能评估
- 多轮追问

如果不拆，最常见结果就是：
- `ask1`
- `ask2`
- `ask3`
- `ask4`

一路复制下去，最后每个脚本都有点像，又都不完全一样，维护成本会炸。

### 3. 难以定位问题出在哪一层
如果检索不准，你会问：
- 是 Milvus 查询的问题？
- 是重排逻辑的问题？
- 是 Prompt 的问题？

如果全写在一起，很难单独验证某一层输出。

### 4. 不利于后续做实验
你后面明明做了很多实验：
- 调权重
- 加噪音过滤
- 加 cross-section support
- 改 prompt 目录化
- 加 session store

这些都说明你已经进入“高频试错”阶段。  
这时候如果没有模块化，实验成本会非常高。

---

**我帮你整理成更标准的答案**

> 版本演进里把 `Retriever`、`PromptBuilder`、`RAGPipeline` 拆开，核心原因是让不同职责的逻辑解耦，方便调试、复用和持续迭代。  
> `Retriever` 负责从 Milvus 取候选证据，解决“取什么”的问题；`PromptBuilder` 负责根据问题类型和模板组织消息，解决“怎么问模型”的问题；`RAGPipeline` 则负责把检索、重排、去噪、上下文组装、模型调用这些在线问答步骤串起来，作为整体编排层。  
> 如果不拆，后面会卡在几个地方：第一，改一个策略就要改整个主流程；第二，新场景只能不断复制脚本；第三，出了问题很难判断是检索、排序还是 Prompt 层的问题；第四，实验和迭代成本会越来越高。  
> 所以拆开并不是为了“写得更抽象”，而是为了让简历 RAG 从一次性脚本，演进成可组合、可实验、可迁移的能力模块。
