## 这题真正的核心

### 为什么会话历史不能直接混进 `resume_profile_chunks`？
因为 `resume_profile_chunks` 里存的应该是：

- **候选人的原始事实知识**
- 也就是简历本身拆出来的结构化证据块

而会话历史存的则是：

- **某个用户在某次问答过程中的上下文状态**
- 包括用户问题、AI 回答、当前 session 的临时语境

这两类数据的职责完全不同。

如果直接混在一起，会带来几个问题：

### 1. 语义污染
AI 回答本身不是原始事实。  
如果把历史对话也写进同一个向量库，后面检索时就可能召回：

- 简历原文
- 以及 AI 之前“说过的话”

这会导致系统越来越像在“引用自己”，而不是引用候选人的真实资料。

### 2. schema 语义混乱
简历块和会话块的数据结构、用途都不一样：

- 简历块强调 `section / subsection / entity_type`
- 会话块更关心 `sessionId / turn / role / createdAt`

混在一个 collection 里，会让 schema 语义越来越乱。

### 3. 调试困难
后面如果回答不准，你会很难判断：
- 检索到的是原始简历事实
- 还是某次历史回答的二次加工内容

这对学习和工程排查都很糟糕。

### 4. 知识库和记忆库边界被打穿
这一点特别重要：

- `resume_profile_chunks` 本质是 **knowledge base**
- 会话历史本质是 **memory / session state**

知识库和记忆库不是一回事。

## 如果以后真要做“历史语义检索”，应该怎么演进？
你这里提到“版本管理”，这是一个很好的补充。  
但在那之前，第一步应该先做的是：

### 演进路径 1：先把会话历史独立出来
不要混进 `resume_profile_chunks`，而是单独存：

- 本地 JSON
- SQLite
- Redis
- 或单独的 session store

也就是说，先把“会话状态”从“简历知识”里分家。

### 演进路径 2：如果只是普通多轮对话，不一定要向量化
很多时候多轮只需要：

- `question + answer`
- `sessionId`
- `createdAt`

按时间顺序存起来就够用了。

因为这时需求只是：
- 接着聊
- 知道上一轮说了什么

不一定需要语义检索。

### 演进路径 3：只有当会话变长、需要按语义回忆时，再考虑独立 history collection
比如以后真要支持：

- “你上次问过什么？”
- “我们之前聊过候选人哪些项目？”
- “跨多轮、多 session 找历史重点”

这时候才适合把历史做成独立向量库，比如：

- `resume_chat_history_chunks`

而不是塞回 `resume_profile_chunks`。

### 演进路径 4：这时你说的版本管理就很重要了
到了历史语义检索阶段，确实应该补这些维度：

- `resumeId`
- `resumeVersion`
- `userId`
- `sessionId`
- `timestamp`

这样才能知道：
- 这段历史是在讨论哪份简历
- 是哪个版本
- 属于谁的会话
- 发生在什么时间点

所以你提到的这部分，其实是**进阶演进层**，方向是对的。

## 我帮你整理成更标准的答案

> 会话历史不能直接混进 `resume_profile_chunks`，因为 `resume_profile_chunks` 存的是候选人的原始简历事实，它是知识库；而会话历史存的是某次问答过程中的上下文状态，它更像记忆或 session 数据。  
> 如果把两者混在一起，会带来语义污染、schema 混乱、调试困难等问题，甚至可能让系统在后续检索时召回 AI 自己以前说过的话，而不是召回原始简历证据。  
> 更合理的做法是先把会话历史独立存储，比如本地文件、SQLite、Redis 或单独的 session store。  
> 只有当会话足够长、确实需要“历史语义检索”时，再考虑单独建立 history collection，例如专门的 `resume_chat_history_chunks`，并补上 `userId`、`sessionId`、`resumeVersion`、`timestamp` 等元信息。  
> 这样才能把“知识库”和“记忆库”的职责分清，也更适合后续版本演进。
