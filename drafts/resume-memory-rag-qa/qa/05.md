## 5. 用户问“这个候选人有哪些 AI Agent 开发相关经验？”，为什么纯向量召回容易先命中 skills，但业务上你又不希望它排第一？

从 查询意图上来说，这句更偏向于要能证明 这句话是否成立，而向量召回的 skills 只能说列出了有哪些技能，但无法形成一个完整的证明链路。 

所以在功能和设计上，我们做了升序降序，把 skills 命中的权重值降低了，把 projects experiences 里的权重适当提高了

**我帮你补完整这题的底层逻辑**

之所以**纯向量召回**容易先命中 `skills`，原因通常有三个：

**1. 问题字面上就和 skills 很像**
用户问的是：

- “AI Agent 开发相关经验”

而简历里恰好有一个技能小节就叫：

- `AI Agent 开发`

从向量相似度角度看，这种“标题高度贴脸”的内容天然容易排前。

**2. skills 表达更短、更集中、更像标准答案**
技能描述通常会写得很集中，比如：

- 高频使用哪些工具
- 搭建过多 Agent 工作流
- 做过 Prompt Engineering

这类句子语义密度高，很容易和 query 对齐。

而项目 / 工作经历的表述常常更分散，可能写成：

- 做了 AI 工作台
- 做了流式回答
- 做了工作流编排
- 做了模板生成

这些虽然更能证明“经验”，但字面不一定直接重复 “AI Agent 开发” 这几个词。

**3. 向量召回默认只管“像不像”，不管“能不能证明”**
纯向量检索本质上更擅长回答：

- “哪些内容和问题语义接近？”

但它不天然懂业务优先级，不会自动知道：

- `skills` 是能力概述
- `projects / work_experience` 才更像实践证据

---

**为什么业务上又不希望它排第一？**
因为这类问题问的是：

- “他有哪些相关经验？”
- 本质更接近“做过什么”“在哪些项目/工作里体现出来”

所以业务上更希望优先看到的是：

- 项目经历
- 工作经历
- 能形成“真实实践证明链”的内容

而不是只看到：

- 技能列表
- 自我概括
- 能力标签

也就是说：

- `skills` 适合做补充证据
- `projects / work_experience` 才应该做主证据

---

**你这题可以整理成一个更标准的答案**

> 纯向量召回容易先命中 `skills`，因为问题里有“AI Agent 开发”这类表达，而简历里的技能小节标题和内容通常会直接包含这些词，语义上和 query 非常接近，所以相似度天然更高。  
> 但业务上我又不希望它排第一，是因为用户问的不是“他会哪些技能”，而更像是在问“他有哪些真实做过的相关经验”。  
> `skills` 更像能力概述，能说明“会什么”；而 `projects` 和 `work_experience` 更能说明“在哪做过、怎么做的、做出了什么结果”，更适合作为主证据。  
> 所以后续优化里，我们不是简单否定 `skills`，而是把它从“主证据”降级为“补充证据”，同时提高 `projects / work_experience` 在经验类问题中的排序权重。
