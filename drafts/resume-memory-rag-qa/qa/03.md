## 3. 为什么“向量维度不能想当然写死”？这个坑在“切换模型供应商”时为什么特别容易出现？

我一开始用的千问 embedding text-3，最大支持 1536维度，然后跟着教程设置了1024，想当然认为向量库都是可以自己随意设。

但是 切换到GLM发现，不同模型厂商有各自的处理，如之前的测试数据，我用的版本最大支持1536维度，但查询和生成只用了256维度，解释是 模型会根据数据自行处理判断，不需要这么大，造成浪费。

后来我加了容错处理，就是不要一开始设死，在查询时，先看返回向量维度，然后根据该值同步设置chunk split 后的向量维度，让两者保持统一。

---

这题你已经抓到主干了，比第 2 题更稳。  
如果按阶段考校，我会给你 `8/10`。

**你答对的核心**
- 你已经意识到：`向量维度不是向量库自己随便定的，而是要和 embedding 模型的真实输出对齐`
- 你也抓到了：这个坑在“切换供应商”时特别容易出现，因为不同厂商的 embedding 接口行为不完全一样
- 你已经做出了正确修正思路：`不要先写死，先探测真实返回维度`

**这里我帮你校正一个小点**
你最后这句：

> 根据该值同步设置chunk split 后的向量维度

这里表述不太准。  
`chunk split` 本身不决定向量维度，`splitter` 只负责切文本。真正要对齐的是：

- collection schema 里的 `vector dim`
- 写入时 embedding 返回的向量维度
- 查询时 query embedding 返回的向量维度

所以更准确的说法应该是：

> 先探测 embedding 模型真实返回的向量维度，再用这个维度创建或校验 Milvus collection，并保证插入和查询两端使用一致的 embedding 维度。

---

**我帮你把这题整理成一个更标准的答案**

> 向量维度不能想当然写死，因为向量库里的 `dim` 不是一个随便填的参数，它必须和 embedding 模型真实输出的向量长度一致。  
> 如果 schema 里写的是 `1024`，但模型实际返回的是 `256`，那写入时就可能失败；即使勉强建好了，查询时也会因为 query embedding 维度不一致而报错。  
> 这个坑在切换模型供应商时特别容易出现，因为不同厂商对 embedding 的支持方式不完全一样：有的模型支持显式指定维度，有的则是模型自己决定输出维度；同时不同平台的 `baseURL`、聊天模型名、向量模型名也往往是分开的。  
> 所以一旦切换供应商，不能默认之前的维度配置还能继续用，而应该先实际调用 embedding 接口，拿到真实返回维度，再用它去创建或校验 collection，并保证“写入”和“查询”两端使用一致的 embedding 配置。
