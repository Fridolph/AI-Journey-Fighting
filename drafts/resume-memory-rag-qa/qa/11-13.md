## 11. 如果一个问题是“他最能代表自己的 AI 项目是什么”，你会怎么定义 primary evidence 和 support evidence？

实际做过的 东西为 primary evidence，如实际项目经验，是如何做的;  其他的 公司经历，skills 等只作为辅助判断；

## 12. 你现在怎么理解“阈值不是通用真理，而是当前数据集上的经验值”这句话？

对于不同的原始数据，不同的回答，计算值可能是不同的，所以我们不能过分依赖主观经验，去 “微调” 参数和数值，而是要建立在不同数据集的 参考体系，数值是不同的基础上，来进行校正与调整，否则就会陷入经验主义误差陷阱（如，这里的 skills 我作了 - 0.08 的阈值调整） 但不意味着 下一个 RAG知识库，做百科全书检索的， 相关联数值也做 -0.08  -0.06 这样的微调，而是要实际根据 milvus collection vector 等，观察实际召回的数据质量来判断如何取用和调整这个值；

**11. primary evidence / support evidence**
这题你答得对路，`8/10`。

你抓到的核心是：
- `primary evidence` 应该优先选“真实做过、能形成证明链”的内容
- `support evidence` 才是公司经历概括、skills、辅助说明

我帮你再收紧一点：

- `primary evidence`
  - 最直接回答问题主语的证据
  - 比如：`my-resume`、`GS`、某段明确的 AI 工作流 / AI 工作台 / Agent 实践项目
  - 要能回答“是什么项目、做了什么、为什么代表他”

- `support evidence`
  - 不直接当主角，但能补强主结论
  - 比如：`skills / AI Agent 开发`、`core_strengths / AI 工程化实践`
  - 用来证明“这不是偶然做过，而是有持续能力和方法论”

这题最关键的一句话是：
- `primary evidence` 负责立主线
- `support evidence` 负责补可信度和信息面

**12. 阈值不是通用真理，而是当前数据集上的经验值**
这题你答得很好，`9/10`。

你已经抓到了本质：
- 不同数据集、不同 embedding 模型、不同 collection 分布，分数区间都可能不一样
- 所以不能把某次实验调出来的阈值，当成通用规则复制到下一个知识库

这就是很成熟的工程意识。

我帮你再提炼成一句更像方法论的话：

> 阈值不是业务真理，而是“当前数据 + 当前模型 + 当前召回分布”下的局部经验参数。

所以调阈值时，真正看的不是“我主观觉得 -0.08 很合适”，而是：
- 当前召回结果里，强相关和弱相关的分数分布
- 误杀率和漏召回率
- 最终答案质量有没有变好

你这里已经明显脱离“经验主义乱调参”阶段了，这点很好。
